### A. VLA 后训练与人机协同纠错

#### 1) Hand-in-the-Loop: Improving Dexterous VLA via Seamless Interventional Correction
- **标题**：Hand-in-the-Loop: Improving Dexterous VLA via Seamless Interventional Correction
- **简称**：HandITL
- **领域**：Dexterous VLA / Interactive Imitation Learning / 人机协同纠错
- **论文**：https://arxiv.org/abs/2605.15157
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：正文未给公开仿真 benchmark，核心评测集中在真实长时序双臂灵巧操作。真机：3 个 long-horizon bimanual dexterous tasks——**Unscrew Assembly With Electric Drill、Unscrew Assembly By Hand、Pick Up and Place the Parts**；并单独做 takeover discontinuity、post-takeover manipulation 与 post-training 对比。设备配置：**2× Franka FR3（双 7-DoF）+ 2× Bytedexter V2（双 21-DoF 灵巧手）**，总计 56-DoF；人类接管侧使用 **Meta Quest 3 controllers + Manus Quantum Metagloves + dual pedals**；感知为 **3 个外置 RGB-D 相机**，视角覆盖 **top-down / bottom-up / third-person**，wrist/head 相机位姿数量为 **0 / 0**。
- **算力**：正文核验到的数据规模为：先采集 **20 小时真实遥操作数据** 训练 base policy，再加入 **1 小时** full-takeover 或 copilot correction 数据做 post-training；额外数据与 teleoperation baseline 保持等时长、等采样比、等训练日程。**GPU 型号、GPU 数量、总训练时长、显存占用正文未披露**。
- **解读**：这篇不是重做一个更大的 VLA，而是把“在线接管纠偏”做成无缝接口，解决高自由度灵巧手 takeover 时的指令跳变。亮点是真机三项长程任务与后训练都涨点；局限是算力与公开代码未披露。复现建议先对齐相对重定向、手臂 residual shared control 和 20h+1h 数据配比。

### B. 统一具身模型与世界-动作联合建模

#### 1) Pelican-Unified 1.0: A Unified Embodied Intelligence Model for Understanding, Reasoning, Imagination and Action
- **标题**：Pelican-Unified 1.0: A Unified Embodied Intelligence Model for Understanding, Reasoning, Imagination and Action
- **简称**：Pelican-Unified 1.0
- **领域**：Embodied Foundation Model / Unified VLM-VLA-World Model
- **论文**：https://arxiv.org/abs/2605.15153
- **项目**：正文未披露
- **代码**：正文、abs 页、PDF 正文当前均未披露
- **评测**：仿真 benchmark：**8 个 VLM benchmark**、**RoboTwin 50-task dual-arm benchmark**、**WorldArena**。正文核验主结果：VLM 平均 **64.7**，WorldArena **66.03 EWM** 排名第一，RoboTwin 平均成功率 **93.5**。真机：**UR5e 机械臂** 与 **Tienkung humanoid robot** 两个平台，重点验证 **compositional generalisation** 与 **zero-shot transfer**；其中 UR5e 任务示例为 **RJ45 插接 + waterproof** 的组合工业面板操作。设备配置：机器人本体明确为 **UR5e** 与 **Tienkung humanoid**；末端执行器型号正文未统一表式披露；视觉侧正文只明确出现 **side-view / top-view observations**，并在动作条件视频示例中出现 **ALOHA-AgileX head camera** 提示，但**真机主实验的相机数量、位姿、wrist/head 拓扑未系统披露**。
- **算力**：正文与附录可核验到训练设置包括：对 5 个 seen tasks 做 joint training，平均 **每任务约 300 个 video-action episodes**；3 个 unseen tasks **每任务 50 个视频序列**用于泛化评估。Unified Future Generator 采用 flow-matching 联训语言、视频、动作损失，但**GPU 型号、GPU 数量、训练时长、总步数、推理显存正文未披露**。
- **解读**：Pelican-Unified 的野心很大：让理解、推理、想象、动作共享一个 backbone 和一套训练环路，而不是串三套专家系统。亮点是 RoboTwin 与 WorldArena 都没掉队，且真机还做了组合泛化；局限是工程细节和算力账单披露不足。复现建议先拆出统一表征 + future video/action 共训练，再验证组合任务收益是否真实来自 imagination 闭环。

### C. 状态估计与接触丰富场景鲁棒性

#### 1) CoCo-InEKF: State Estimation with Learned Contact Covariances in Dynamic, Contact-Rich Scenarios
- **标题**：CoCo-InEKF: State Estimation with Learned Contact Covariances in Dynamic, Contact-Rich Scenarios
- **简称**：CoCo-InEKF
- **领域**：Legged State Estimation / Contact-Rich Robotics / Learned Covariance
- **论文**：https://arxiv.org/abs/2605.15122
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：正文以自建动态运动仿真为主，覆盖 **100 条 diverse dance motion trajectories** 的一致性与误差评估，并在策略闭环中测试多类动态动作；未采用单独公开具身 benchmark。真机：**Lima legged robot** 两类实验——(1) **20 条 messy ground-motion sequences** 的离线 MoCap 对照；(2) 配合 **VMP controller** 做 **13 个动态舞蹈动作** 的 in-the-loop 实测，并额外包含 **pirouette、moonwalk** 两个未见动作。设备配置：机器人本体为 **Lima**；末端执行器为 **四足足端接触点**（ground motions 取 **10 个 contact points**）；论文明确强调**不使用 camera / LiDAR 等外感知**，主要依赖 **IMU + 接触信息**；相机位姿数量 **0**，wrist/head 相机 **不适用**。
- **算力**：正文/附录核验：学习模型在 **单张 NVIDIA RTX 4090** 上训练 **100k iterations**，最长 **约 5 天**；并使用 **E = 1280 parallel environments**、历史长度 **H = 20**、buffer 长度 **L = 128**；所有仿真在 **600 Hz** 下运行。实时基准为在机器人板载计算机上单线程执行，部分 baseline 因算力不足无法在 **600 Hz** 真机回路内运行。
- **解读**：CoCo-InEKF 把二值接触状态升级为“可学习的连续接触协方差”，本质是在滤波器里显式表达部分接触和方向性打滑。亮点是真机 Lima 舞蹈动作里也能跑进 600Hz 回路；局限是任务更偏状态估计而非高层具身决策。复现建议优先对齐 contact candidate 设计、4090 上的 100k 训练设置以及 600Hz 实时约束。

### D. Flow-Matching 闭环控制

#### 1) Learning Direct Control Policies with Flow Matching for Autonomous Driving
- **标题**：Learning Direct Control Policies with Flow Matching for Autonomous Driving
- **简称**：Direct Control Flow Matching
- **领域**：Flow Matching / Autonomous Driving / Closed-loop Control
- **论文**：https://arxiv.org/abs/2605.14832
- **项目**：https://marcelloceresini.github.io/DirectControlFlowMatching/
- **代码**：项目页当前未见公开代码仓链接
- **评测**：仿真 benchmark：论文主评测为**自建闭环 reactive-agent driving simulator**，指标对齐 **nuPlan / NAVSIM** 风格，报告 **collision rate、drivable area compliance、route progress、predicted/executed jerk**；训练数据来自 **2,524,298 帧**城市道路与环岛场景。真机：正文展示目标部署平台为 **Lexus RX 350h** 实车，并说明动力学模型拟合真实测试车；但**主结果仍为闭环仿真，未给公开实车定量表**。设备配置：机器人本体为 **Lexus RX 350h**；末端执行器不适用；正文仅说明车载有 **cameras、radars、GPS**，**传感器数量、相机位姿数量正文未披露**。
- **算力**：正文/附录核验：BEV encoder 与 vector-field U-Net 在 **单张 NVIDIA RTX A6000** 上从零联合训练，**300k steps**、batch size **32**，约 **3 天**完成；推理端在 **NFE=10** 时总前向时间约 **25 ms**，满足闭环 **20 Hz** 重规划时延预算（同样以 **RTX A6000** 为参考）。
- **解读**：这篇价值在于把 flow matching 直接下放到可执行控制量，绕开先规划轨迹再做控制跟踪的两段式链路。亮点是 20Hz 闭环与低 NFE 延迟比较扎实；局限是公开主结果仍停留在仿真、真实车仅作部署指向。复现建议严格保持 BEV 表示、reactive-agent 闭环评测和 NFE=10 的延迟设定。

## Agent / OpenClaw 能力增强（固定栏目）

### 今日新增 / 值得关注

- **OpenDeepThink** 和 **Orchard** 也进入了候选池，说明“并行推理 + agent 工作流基础设施”还在快速迭代，后续若出现可核验开源实现，适合单列成 agent 工具专题。

### 能力提升方法 / 工作流要点
- 今天这批论文对 OpenClaw 类 agent 的直接启发有三点：
  1. **在线纠偏接口要先于大模型扩容**：HandITL 说明长期执行失败往往来自中途偏差累积，而不是初始规划不够聪明。
  2. **统一表征比多模块拼接更值得投资源**：Pelican-Unified 虽然还不够透明，但它把 reasoning / imagination / action 放进同一训练环路，这比单纯接 API 串流水更像长期方向。
  3. **闭环评测必须把实时性写进主指标**：无论是 CoCo-InEKF 的 600Hz，还是 Direct Control Flow Matching 的 20Hz/NFE=10，真正能上机器人或设备的系统都在正面回答 latency budget。

### 机器人 / 设备控制能力相关进展
- **HandITL** 对“agent 远程接管真实双臂灵巧手”很有参考价值：如果 OpenClaw 未来要做 shared autonomy，这类相对重定向与 copilot residual interface 值得优先借鉴。
- **CoCo-InEKF** 提醒一个常被忽略的现实：很多控制失败其实不是策略错，而是状态估计和接触置信度不稳。对真实机器人，感知-估计链路就是策略的一部分。

### 持续监控主题
- Dexterous VLA 的 intervention learning / correction data post-training
- Unified embodied foundation model：共同表征、future imagination、动作联合生成
- 实时 state estimation 与 contact-aware filtering
- Flow-matching / diffusion 在真实闭环控制里的 NFE、延迟与部署预算
