## 具身操作与分层 VLA

### 1. HiVLA: A Visual-Grounded-Centric Hierarchical Embodied Manipulation System
- **简称**：HiVLA
- **领域**：具身操作 / 分层 VLA / 视觉指向操作
- **论文**：<https://arxiv.org/abs/2604.14125>
- **项目**：<https://tianshuoy.github.io/HiVLA-page/>
- **代码**：正文与项目页暂未给出独立代码仓库（项目页仅给出演示与论文链接）
- **评测**：仿真采用 RoboTwin 2.0 的 9 个任务（4 easy + 5 hard），Aloha-Agilex-1.0 双臂平台设定，头部相机 1 个（1920×1080）+ 腕部相机 2 个（720p）；每任务 100 次 unseen 配置试验。真机同样部署在 Aloha-Agilex-1.0 上，覆盖 Click Bell / Pick & Place Cup / Pick & Place Block 三类场景，共 7 个物体类别、16 个子场景，文中未单列末端型号，按平台自带双夹爪执行；相机位姿数量与仿真一致为头部 1 + 腕部 2。
- **算力**：RoboTwin 主训练为 2×NVIDIA H200，batch size 64，150k steps；VLM planner 微调与真机 80k steps 微调的额外算力正文未完整披露。
- **解读**：核心是把高层语义规划与低层动作扩散显式解耦：VLM 先输出“子任务 + 目标框”，DiT 再结合全局视野、局部高分辨率 crop 与 skill token 执行。优点是显著提升长程多阶段任务与多目标消歧；局限是仍依赖规划质量与检测框质量。复现上建议先搭建高分辨率多视角数据与自动框标注链路，再复做 planner/action expert 两阶段训练。

### 3. UMI-3D: Extending Universal Manipulation Interface from Vision-Limited to 3D Spatial Perception
- **简称**：UMI-3D
- **领域**：机器人数据采集 / LiDAR-视觉融合 / 模仿学习基础设施
- **论文**：<https://arxiv.org/abs/2604.14089>
- **项目**：<https://umi-3d.github.io/>
- **代码**：<https://github.com/hku-mars/UMI-3D>；硬件：<https://github.com/Physical-Intelligence-Laboratory/UMI-3D-Hardware>
- **评测**：全部为真机评测。系统是单臂 UMI-3D 手持示教与部署链路，论文正文明确含 fisheye 相机 1、LiDAR 1、IMU 1；机器人本体支持兼容机械臂，文中展示为单臂并联平行夹爪方案，但未在评测段逐项固定某一商品臂型号。任务包括 Cup Arrangement（64 物体组合×10 次=640 次）、Curtain Pulling（120 次）、Door Opening + Cup Placement（40 次）、跨 embodiment Mouse Placement（16 组合×5 次=80 次）。仿真 benchmark：无；真机设备配置：单臂机械臂 + 平行夹爪 + 1 个腕部广角/鱼眼相机，位姿数量 1。
- **算力**：正文未披露训练 GPU 型号；仅披露训练数据规模与编码器，如 CLIP ViT-L/14、DINOv2 ViT-L/14。
- **解读**：UMI-3D 的关键价值不在新 policy，而在把原先依赖单目视觉 SLAM 的 UMI 升级到 LiDAR-centric SLAM，使示教轨迹在低纹理、遮挡、动态场景下更稳。它直接扩展了可采集任务分布，尤其是窗帘、门柜等过去难采的任务。局限是系统更重、仍主要单臂。复现建议先复现时间同步与 LiDAR-camera 外参标定，否则后续 replay buffer 对齐质量会明显下降。

- **领域**：移动操作 / 3D 空间记忆 / 长程任务执行
- **论文**：<https://arxiv.org/abs/2604.13633>
- **项目**：正文未给出
- **代码**：正文未给出
- **评测**：主评测为 ALFRED。测试集包含 seen / unseen 环境，论文报告 step-by-step 指令下 test seen 65.09%、test unseen 60.79%，无详细指导下仍有 61.24% / 56.04%。ALFRED 为仿真 benchmark；文中附录说明 agent 具备移动、旋转与 1-DoF camera tilt，初始相机俯角 45°，视场角 60°。真机实验：无，因此**真机设备配置：无**。

## Agent / OpenClaw 能力增强（新增固定栏目）

> 本栏不追求“论文式 novelty”，而是追踪当下真正能提升研究助手与具身系统生产力的 agent skill、工作流、部署技巧与能力增强路径。优先覆盖：信息搜集、自助科研、机器人/设备控制三类能力。

### 7. OpenClaw 生态中当前值得重点跟踪的高价值 skill / 能力模块
- **ClawHub**：用于搜索、安装、更新和发布 skill，是 OpenClaw 技能生态的“包管理入口”。如果后续想持续扩充日报里的 agent 工具栏，ClawHub 应作为第一监控源。
- **Summarize**：适合把 URL、播客、长视频、长文快速转成结构化摘要，是提升“信息搜集 + 快速初筛”效率的高杠杆技能。
- **GitHub / gh-issues**：前者负责 issue / PR / CI / review 等日常仓库操作，后者更适合把“读 issue → 拉子代理实现 → 开 PR → 跟 review”串成闭环，属于自助科研和自助开发的关键基建。
- **Skill-Creator**：当现成 skill 不够用时，可快速沉淀出新的 AgentSkill，把一次性脚本/流程升级成可复用能力。
- **OpenAI-Whisper / Video-Frames**：一个负责语音转文本，一个负责视频帧抽取，适合把会议、演示、机器人实验录像转成可检索材料。
- **Feishu 系列 skill（doc/task/wiki/perm/drive/urgent）**：偏组织协作层，适合把 agent 从“单机助手”提升为“文档-任务-知识库-提醒”协同节点。

### 8. 提升 OpenClaw 信息搜集能力：三条最有效的方法
- **方法一：把“检索”和“阅读”拆开**。先广泛搜集候选（论文、代码仓、issue、demo、视频、blog），再让 agent 分层阅读：标题/摘要筛选 → 正文核验 → 代码/项目页交叉验证。这样比一步到位问大模型更稳。
- **方法二：优先让 skill 接管专门数据源**。例如 GitHub 问题用 `github`/`gh-issues`，长文/视频用 `summarize`，语音资料用 `openai-whisper`。专用 skill 比通用聊天式问答更稳定，也更容易形成可复用流程。
- **方法三：用 memory 维护长期主题追踪**。把持续关注的话题（如 VLA、RL、OpenClaw、机器人执行链路）写入长期记忆或日报偏好，agent 才能在后续日报里做持续增量跟踪，而不是每次从零开始。

### 9. 提升 OpenClaw 自助科研能力：从“会回答”升级到“会推进任务”
- **任务拆分**：把科研任务拆成检索、筛选、精读、对比、复现实验、整理结论六步，比让一个 agent 一次性产出完整结论更可靠。
- **子代理并行**：复杂问题可拆给多个 sub-agent / ACP harness 分头处理，例如一个看论文方法，一个看代码可复现性，一个看实验与算力，再由主代理合并。
- **事实闭环**：对论文类内容，应该形成“摘要页/二手解读 → PDF 正文 → 项目页/代码仓”的三段式核验闭环，避免只看社交媒体二次传播。
- **沉淀为 skill**：真正长期重复的科研动作，例如“从某批关键词中抓取论文 + 提取项目页 + 归档摘要”，最好做成 skill 或固定脚本，而不是每次重新提示。

### 10. 提升 OpenClaw 机器人与设备控制能力：关键不在模型，而在工具链闭环
- **先做可观测，再做可控**：机器人和设备接入里，第一步不是“让 agent 发控制命令”，而是先让它稳定读到传感器、日志、状态、视频流与错误信息。没有观测闭环，控制只会变成盲操。
- **把控制封装为稳定接口**：对于 ROS2、串口、电源控制器、相机、机械臂等，应该优先提供明确命令入口或 skill，而不是让 agent 即席拼 shell 命令。接口越稳定，越适合长期自动化。
- **采用“感知-决策-执行-验证”闭环**：例如 agent 先读相机/状态，再决定动作，再执行控制，再读取结果确认是否成功。机器人控制的本质不是一次命令，而是闭环纠错。
- **真实机器人能力增强方向**：后续重点值得追踪的包括：ROS2 skill、设备管理 skill、相机/传感器接入 skill、实验录像解析 skill、机械臂任务编排 skill，以及基于 VLA/RL policy 的高层指令接口。
