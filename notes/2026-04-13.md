## 主清单（单一主清单，按技术重点分类）

### A. 生成式具身数据与世界-动作模型

#### 1) VAG: Dual-Stream Video-Action Generation for Embodied Data Synthesis
- **标题**：VAG: Dual-Stream Video-Action Generation for Embodied Data Synthesis
- **简称**：VAG
- **领域**：Embodied Data Synthesis / World-Action Model / VLA 数据生成
- **论文**：https://arxiv.org/abs/2604.09330
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：**LIBERO**（Object / Spatial / Goal / Long），文中给出 action replay 成功率；以 **400 对 video-action** 训练、**50 对**测试，并在 **20,000 iterations** 下评估。真机：一部分是 **AgiBot G1 双臂人形机器人** 数据集上的现实世界评估；另一部分是 **Agilex Cobot Magic 双臂机器人** 的真实回放与下游 VLA 训练验证。设备配置：AgiBot/Agilex 均以**头部相机 head camera** 为主；LIBERO 使用 **head + wrist stitched videos**；Agilex 为双臂夹爪操作，wrist/head 精确位姿数量正文未完整统一披露。下游真机 tableware pick-and-place 中，20 次试验里用 VAG 合成数据预训练后成功率由 **35% 提升到 55%**。
- **算力**：正文披露 VAG 推理采用 **35 denoising steps**；训练环境给出 **NVIDIA H20 GPUs**、**batch size = 1 / GPU**。但**GPU 卡数、总训练时长、统一显存/吞吐账单正文未系统披露**。
- **解读**：VAG 的关键价值是把视频生成和动作生成做成同步双流去噪，避免“先出视频再反推动作”的错位与误差累积。它对合成可训练数据很实用，且已展示真机泛化增益。局限是代码未放、硬件账单不完整。复现建议先照论文在 LIBERO 与小规模双臂真机数据上验证视频-动作对齐，再评估其作为预训练数据源的收益。

#### 2) V-CAGE: Vision-Closed-Loop Agentic Generation Engine for Robotic Manipulation
- **标题**：V-CAGE: Vision-Closed-Loop Agentic Generation Engine for Robotic Manipulation
- **简称**：V-CAGE
- **领域**：Agentic Data Engine / 长时程操作数据合成 / Sim2Real
- **论文**：https://arxiv.org/abs/2604.09036
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真侧基于 **SAPIEN** 构建长时程任务数据，生成 **AutoCheckout、PackBreads、PackStationery、SortToCabinet** 四类任务，每任务 **100 条 expert trajectories**，并在 π0.5 VLA 上做 **100 trials / task** 评测；微调后成功率分别达到 **54% / 54% / 100% / 25%**。真机：**ALOHA-AgileX** 平台执行“**抓取 4 个方块并放入目标盘**”的 Sim2Real 实验。设备配置：**ALOHA-AgileX 双臂本体 + 双夹爪末端**；正文图注与段落明确平台，但**相机数量与 head/wrist 位姿数量未形成统一配置表**。真实评测中，**10 real + 250 sim** 共训优于 **10 real only**，20 次试验成功率由 **20% 提升到 55%**。
- **算力**：正文未披露训练 GPU 型号/卡数/总时长；仅披露视频压缩模块在满足 **<0.1 JOD** 感知阈值下可实现 **93% 以上**文件体积压缩。VLA 训练用 π0.5，但完整算力账单**正文未披露**。
- **解读**：V-CAGE 的贡献不只是“造数据”，而是把**语义布局生成、可执行子任务检索、视觉闭环验收**串成 agentic 合成流水线，专门减少长时程任务里的静默失败。局限是系统更像工程框架，公开代码与算力细节都缺。复现建议优先复刻其 IGSC + 视觉 critic 两个核心模块，再逐步接入 VLA 微调闭环。

### B. 空间推理与装配智能

#### 3) AssemLM: Spatial Reasoning Multimodal Large Language Models for Robotic Assembly
- **标题**：AssemLM: Spatial Reasoning Multimodal Large Language Models for Robotic Assembly
- **简称**：AssemLM
- **领域**：Robotic Assembly / Spatial Reasoning MLLM / 6D Pose Prediction
- **论文**：https://arxiv.org/abs/2604.08983
- **项目**：https://assemlmhome.github.io/
- **代码**：https://github.com/TeleHuman/AssemLM
- **评测**：仿真/离线 benchmark：**AssemBench**，含 **90 万+ multimodal samples、15 万+ assembly steps**，覆盖日常物体、家具与碎片装配等多类别 6D pose 推理。真机：**Flexiv Rizon 4s** 机械臂执行 **Insert Plug、Store Cans、Insert Flower、Build Blocks** 四类真实装配任务。设备配置：机器人本体为 **Flexiv Rizon 4s**；末端执行器类型与相机位姿数量在主文未统一列出；正文明确输入使用 point clouds、manuals 与 instructions，但**真实相机/传感器拓扑正文未给统一设备表**。所有模型在真机部分均以 **30 个 randomized initializations** 评估成功率。
- **算力**：附录披露多模型比较采用统一训练环境：**4× A100/H100 80GB**，**per_gpu_batch_size=4**；附加基线在同一 **130K training samples、12 epochs** 下训练。真实资产微调部分披露 **330 epochs、batch size 4**，但未细分每阶段 wall-clock 时间。
- **解读**：AssemLM 把装配说明书、点云和语言指令统一进一个显式 3D 空间推理框架，目标不是泛泛“看懂场景”，而是输出可执行的任务关键 6D 位姿。亮点是 AssemBench 规模够大，真机装配也做到了。局限是现实传感器细节披露不足。复现建议先跑合成集上的 pose reasoning，再单独做真实资产微调与容差敏感任务验证。

### C. 世界模型与强化学习迁移

#### 4) WOMBET: World Model-based Experience Transfer for Robust and Sample-efficient Reinforcement Learning
- **标题**：WOMBET: World Model-based Experience Transfer for Robust and Sample-efficient Reinforcement Learning
- **简称**：WOMBET
- **领域**：World Model / Offline-to-Online RL / Experience Transfer
- **论文**：https://arxiv.org/abs/2604.08958
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：正文明确在 **MuJoCo continuous control benchmarks** 上验证，比较 **SAC / PPO / TD3** 从零在线训练、offline RL、以及 offline-to-online baseline。论文关注“源任务生成可迁移经验 + 目标任务在线微调”的样本效率与最终性能。真机：**未报告标准化真机实验**，因此机器人本体、末端与相机配置不适用。
- **算力**：正文给出方法层关键设定：用**不确定性惩罚 MPC** 在源任务生成数据，再做自适应 offline/online 混合采样；但**GPU 型号、卡数、batch size、总训练时长正文未披露**。
- **解读**：WOMBET 有意思的点在于不把离线数据当“天降固定资产”，而是用世界模型先生成、筛掉高不确定轨迹，再把可靠经验转给目标任务。理论与实验闭环都完整。局限是仍停留在 MuJoCo，离真实机器人还有落差。复现建议先复刻 uncertainty-penalized MPC + dual-criterion filtering，再看迁移收益对源/目标错配有多敏感。

### D. 灵巧手评测与基准建设

#### 5) A Benchmark of Dexterity for Anthropomorphic Robotic Hands
- **标题**：A Benchmark of Dexterity for Anthropomorphic Robotic Hands
- **简称**：POMDAR Benchmark
- **领域**：Dexterous Manipulation Benchmark / Anthropomorphic Hand Evaluation
- **论文**：https://arxiv.org/abs/2604.09294
- **项目**：https://srl-ethz.github.io/POMDAR/
- **代码**：正文未披露独立代码仓库（项目页说明公开提供 CAD 与 simulation files）
- **评测**：仿真 benchmark：论文将 **POMDAR** 同时实现于**仿真与真实系统**，包含 **vertical、horizontal、continuous rotation、pure grasping** 四类配置、共 **18 个 dexterous tasks**。真机：使用 **ORCA 系列拟人手**（文中展示 **2 / 3 / 5 / 16 DoF** 等 embodiment）进行遥操作评测，并与**人类基线**对照。设备配置：机器人本体/末端即 **ORCA anthropomorphic hands**；该 benchmark 设计上通过机械支架尽量压制 arm/wrist 补偿，因此主比较对象是手本体 dexterity；**相机位姿数量不是主评测变量，正文未给统一相机配置表**。
- **算力**：该工作核心是 benchmark 与实体/仿真评测框架，**训练算力不适用**；正文未提供需要记录的统一训练 GPU 账单。
- **解读**：这篇最重要的是把“灵巧”从泛化口号压缩成可量化 throughput 指标，并把真实任务谱系、3D 可打印装置、仿真对应物统一起来。局限是它评的是手部 dexterity，不直接等价于全系统操作能力。复现建议优先按项目页打印标准件并在仿真/真机双侧对齐评分流程，先建立跨手型可比较基线。
