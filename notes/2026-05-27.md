### A. VLA 数据对齐、可控执行与持续学习

#### 1) FineVLA: Fine-Grained Instruction Alignment for Steerable Vision-Language-Action Policies
- **标题**：FineVLA: Fine-Grained Instruction Alignment for Steerable Vision-Language-Action Policies
- **简称**：FineVLA
- **领域**：VLA / 细粒度指令对齐 / Steerable Control
- **论文**：https://arxiv.org/abs/2605.27284
- **项目**：https://finevla.xlang.ai/
- **代码**：https://github.com/xlang-ai/FineVLA
- **评测**：仿真 benchmark：RoboFine-Bench（500 videos、10,816 atomic facts、1,030 VQA questions）+ RoboTwin / AlohaMix-OFT；其中 RoboFine-VLM 在 hard 设定下做到 **71.0% VQA / 83.6% caption**，策略在 AlohaMix-OFT Easy/Hard 达到 **86.8% / 82.5%**。真机：Cobot Magic 双臂平台上的 instruction-sensitive 操作评测，FG:Raw=1:1 策略平均 **62.7/100**（Raw-only 为 49.9）。设备配置：**Cobot Magic 双臂机器人**；末端执行器与外置相机型号正文未明确；正文可确认多视角设置，真实平台相机位姿数量**正文未披露**。
- **算力**：正文/附录披露较完整：策略预训练 **100k steps，64×A100，per-device batch 8，global batch 512**；RoboTwin 微调 **100k steps，8×A100，global batch 128**。RoboFine-VLM SFT 另披露 **256×H200、global batch 512、9 epochs、约 40 小时**。真实平台微调使用 **12 个桌面任务、每任务 50 条示教，共 600 demonstrations**。
- **解读**：这篇把“做什么”扩展成“怎么做”，核心价值是把 active arm、接触区域、接近方向这类执行细节真正写进监督信号。亮点是真机 steerability 提升明确；局限是硬件细节披露不完整。复现建议先复做 FG:Raw 配比曲线，再看增益究竟来自更细语言还是更强数据筛选。

#### 2) Can VLA Models Learn from Real-World Data Continually without Forgetting?
- **标题**：Can VLA Models Learn from Real-World Data Continually without Forgetting?
- **简称**：ContinualVLA
- **领域**：VLA / Continual Learning / Real-World Manipulation
- **论文**：https://arxiv.org/abs/2605.26820
- **项目**：https://agentic-intelligence-lab.org/Never
- **代码**：https://github.com/Agentic-Intelligence-Lab/ContinualVLA
- **评测**：仿真 benchmark：无，本文专注真实持续学习。真机：AgileX PiPER 单臂平台，4 个顺序任务组成真实 continual stream：Stack Bowl、Hang Cup、Press Button、Fold Towel。设备配置：**AgileX PiPER（6-DoF arm）+ 1-DoF gripper**；视觉为 **2 个 wrist-mounted cameras + 2 个 external cameras**，共 **4 个 RGB 机位**，分辨率 480×640、30Hz。结果上，naive sequential FT 在最终评测时对早期任务出现明显遗忘，例如 D1/D2/D3 仅 **15.0 / 25.0 / 13.3**；经验回放显著缓解遗忘，且优于 joint training。
- **算力**：基座策略为 **π0.5**；SFT 配置为 **AdamW + cosine decay，batch size 128，action horizon 10，每任务 4,000 steps**。GPU 型号、卡数、总训练时长正文未披露。
- **解读**：价值在于它把 continual VLA 从“仿真里看起来能行”拉到真实异构任务流里，直接暴露灾难性遗忘。亮点是真实任务设计跨刚体、接触和可形变物体；局限是任务规模仍小。复现建议优先复现 replay buffer 大小与 replay frequency 的 U 型曲线，而不是只报最终平均分。

### B. Sim2Real 流水线与操作中间表征评测

#### 1) HyperSim: A Holistic Sim-To-Real Framework For Robust Robotic Manipulation
- **标题**：HyperSim: A Holistic Sim-To-Real Framework For Robust Robotic Manipulation
- **简称**：HyperSim
- **领域**：Sim2Real / Robotic Manipulation / Synthetic Data
- **论文**：https://arxiv.org/abs/2605.26638
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：基于 **O3DE** 构建高保真合成环境，并兼容 IsaacSim / Sapien；对 ACT 与 π0 两类策略做 zero-shot/few-shot sim2real 研究。真机：总计 **400+ real-world trials**，以 **TAR / SR1 / SR3** 三项细粒度指标评测；完整管线在真实执行上达到 **ACT 80%**、**π0 95%** 的 sim2real success rate，并在物理扰动下带来 **35% 更高 completion rate**。设备配置：正文明确存在 **标准两指夹爪**、**wrist camera**，背景重建使用 **RGB + LiDAR + IMU**；但机器人本体型号、真实相机总数量与固定位姿数量**正文未披露**。
- **算力**：正文披露数据规模与流程，但未给统一 GPU 型号/卡数/训练时长账单；仅说明真实侧只需 **一次环境扫描 + 数十条示教** 即可启动整套流程，故算力项记为 **正文未披露**。
- **解读**：HyperSim不是单点提分技巧，而是把高保真场景重建、对抗式轨迹生成和 sim-real 共训串成闭环。亮点是真实 400+ 次评测和扰动鲁棒性结果都够硬；局限是训练算力与具体硬件公开不足。复现建议先单拆 adversarial trajectory 模块，验证鲁棒增益是否独立成立。

#### 2) OSMa-Bench++: Toward Open-Ended Benchmarking of Semantic Mapping for Manipulation with Prompt-Generated Synthetic Scenes
- **标题**：OSMa-Bench++: Toward Open-Ended Benchmarking of Semantic Mapping for Manipulation with Prompt-Generated Synthetic Scenes
- **简称**：OSMa-Bench++
- **领域**：Semantic Mapping / Manipulation Benchmark / Synthetic Scene Generation
- **论文**：https://arxiv.org/abs/2605.26831
- **项目**：正文未披露
- **代码**：https://github.com/be2rlab/OSMa-Bench-v2
- **评测**：仿真 benchmark：基于 **SceneSmith + Habitat/Habitat-Lab + HaDaGe + OSMa-Bench** 的扩展评测流水线，生成 RGB-D 观测序列、3D map、scene graph completeness 与 VQA 评测；新增 **PromptGT** 问答分支，专门 stress-test clutter、small objects、partial occlusions、lighting variation。真机：**无真实机器人实验**。设备配置：不适用；仿真输入可确认包含 **RGB-D observation sequences + camera poses**，真实机器人本体/末端/机位数不适用。
- **算力**：正文主要披露流程组件与数据生成链路，未给 GPU 型号、卡数、时长或统一成本表，按要求记为 **正文未披露**。
- **解读**：这篇的意义在于把“语义地图是否真能服务操作”从静态封闭数据集里解放出来，用 prompt-controlled 场景专打操控角落案例。亮点是 PromptGT 让问题生成与场景意图对齐；局限是还停在仿真。复现建议把同一 mapping 方法在原版 OSMa 与 PromptGT 设定下并排测，才能看出是否真的对操作相关语义更敏感。

### C. 具身工具化与 Agent 能力外化

#### 1) Enabling Extensible Embodied Capabilities with Tools
- **标题**：Enabling Extensible Embodied Capabilities with Tools
- **简称**：ETP
- **领域**：Embodied Agent / Tool Use / Modular Capabilities
- **论文**：https://arxiv.org/abs/2605.26637
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：EmbodiedBench 的 **EB-ALFRED / EB-Habitat / EB-Navigation / EB-Manipulation**，并额外构造 EmbodiedToolBench 覆盖 tool-need recognition、tool selection、tool execution、tool-chain composition 四类能力；工具增强平均带来 **EB-ALFRED +31%**、**EB-Navigation +36%**。真机：桌面清理、天平配重、积木搭建三类真实任务，每类 **10 次**；无工具时各模型均 **0/10**，有工具后如 GPT-4o 在 desktop cleaning **8/10**、Claude-4.6 在 balance scale **9/10**。设备配置：正文可确认 **Franka Panda + parallel gripper**（EB-Manipulation 提示模板明确）；真实实验相机位姿数量与末端附加传感器型号**正文未披露**。
- **算力**：评测含 API 成本约 **3000 美元**；本地部署使用 **8× NVIDIA A800 GPUs** 进行工具推理与开源组件部署。训练算力未统一披露。
- **解读**：这篇很像给具身系统做“OpenClaw 化”：把感知、规划、控制拆成可注册、可调用、可组合的工具层。亮点是真机无工具全灭、有工具显著起量；局限是 execution-type tool 仍然弱。复现建议不要只测最终成功率，要单独拆 need/selection/execution/chain 四段误差。

#### 2) MUSE-Autoskill: Self-Evolving Agents via Skill Creation, Memory, Management, and Evaluation
- **标题**：MUSE-Autoskill: Self-Evolving Agents via Skill Creation, Memory, Management, and Evaluation
- **简称**：MUSE-Autoskill
- **领域**：LLM Agent / Skill Lifecycle / Memory
- **论文**：https://arxiv.org/abs/2605.27366
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真/模拟 benchmark：**SkillsBench**，共 **51 个 real-world-style tasks**，由标准化 **Docker** 环境中的自动 verifier 评分，覆盖 science & engineering、data analysis、document processing、ops & planning 四大域。真机：**无真实机器人或物理设备实验**。设备配置：不适用。结果上，MUSE-Autoskill 在带 human skills 设定下总体达到 **68.4%**，较自身无技能基线提升 **15.21 个百分点**；在 35 个成功生成技能的任务上准确率达 **87.94%**。
- **算力**：正文未披露统一训练 GPU；实验主体为 **GPT-5.5-backed agents** 在 Docker 任务环境中运行，更多是 API/运行时消耗而非传统集中训练账单，因此算力项按要求记为 **正文未披露**。
- **解读**：它把 skill 从一次性 prompt 产物升级成“能创建、会测试、可回收、可再训练”的长期资产，这对 OpenClaw/agent 工程非常对路。亮点是 skill-level memory 真正进入闭环；局限是基准仍偏软件任务。复现建议优先把 evaluator 和 unit test 体系搭牢，否则技能库很快会被脏 skill 污染。

#### 3) VitaBench 2.0: Evaluating Personalized and Proactive Agents in Long-Term User Interactions
- **标题**：VitaBench 2.0: Evaluating Personalized and Proactive Agents in Long-Term User Interactions
- **简称**：VitaBench 2.0
- **领域**：Agent Benchmark / Personalization / Proactive Interaction
- **论文**：https://arxiv.org/abs/2605.27141
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真/模拟 benchmark：VitaBench 2.0 以 **56 users、66 tools、2000+ preferences、819 tasks** 构成长时任务序列，场景覆盖 Delivery / In-store Consumption / OTA；任务同时看 **trajectory-level** 与 **outcome-level** rubric，并支持 Full Context、Agentic Memory、RAG Memory 三类记忆接口对比。真机：**无真实机器人/真实设备实验**；全文明确为 controlled simulation。设备配置：不适用。
- **算力**：作者说明评测主要走商业 API，**无高强度本地 CPU/GPU 需求**；评测限流为 **200 RPM**、最多 **20 个并发 user-agent rollouts**。集中训练 GPU 账单正文未披露。
- **解读**：这篇把“会不会做事”推进到“能不能在长时互动里持续理解同一个人”，很适合检验 agent 的记忆工程是否真有用。亮点是把 personalization 和 proactiveness 放在同一基准里；局限是仍是模拟用户。复现建议重点比较 Full Context、structured memory、RAG memory 三种接口，而不是只换底模。

## Agent / OpenClaw 能力增强（固定栏目）

### 今日增量观察
- **FineVLA + ContinualVLA** 共同说明：VLA 下一阶段不只是更大会做，而是更能被细粒度指挥、也更能长期在线学习。
- **ETP / MUSE-Autoskill / VitaBench 2.0** 形成了一条很清晰的 agent 工程链：工具外化 → 技能资产化 → 长时个性化评测。
- **OSMa-Bench++** 值得当作 OpenClaw 接视觉-操作系统时的中间表示诊断基线：别只测末端成功率，也要测地图语义是否真的可问可用。

### 对 OpenClaw / skill 工作流的直接启发
1. **skill 不该只是文档**：MUSE-Autoskill 证明 skill 最终要有测试、记忆和版本演化，才能越用越强。
2. **工具调用评测要分段**：ETP 把 need / selection / execution / chain 拆开，非常适合照搬到 OpenClaw skill 调用质检。
3. **长期用户记忆要基准化**：VitaBench 2.0 说明“记住用户”不能只靠主观感觉，必须有长时序任务与 preference shift 评测。
