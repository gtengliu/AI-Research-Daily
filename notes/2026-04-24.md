### A. 长时程 VLA 规划与跨视角操作

#### 1) Long-Horizon Manipulation via Trace-Conditioned VLA Planning
- **标题**：Long-Horizon Manipulation via Trace-Conditioned VLA Planning
- **简称**：LoHo-Manip
- **领域**：VLA / 长时程任务规划 / 具身操作
- **论文**：https://arxiv.org/abs/2604.21924
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：RoboVQA、EgoPlan-Bench2、EmbodiedBench（EB-Alfred、EB-Habitat）、ShareRobot-T、VABench-V、LIBERO、VLABench。真机：Franka 长时程桌面操作。设备配置：Franka 机械臂 + gripper；2 路 Intel RealSense，相机位姿为 1 路俯视 top-view + 1 路 wrist-view。
- **算力**：部署侧正文披露：在 NVIDIA A6000 上，planner 约 2 Hz、低层 VLA executor 约 10 Hz；manager 每 100 个低层控制步重规划一次。训练资源正文/附录未给出完整 GPU 卡数、总时长与 batch 账单，记为“正文未完整披露”。
- **解读**：这篇把长时程管理器与低层 VLA 解耦，用“已完成/未完成子任务 + 2D 轨迹 trace”做轻量显式记忆，核心优点是跨执行器、跨 embodiment 更容易迁移。局限是完整训练算力未细报，且真实数据主要来自 Bridge 子集；复现建议优先搭一版“manager 重规划 + trace 条件控制”最小闭环。

#### 2) VistaBot: View-Robust Robot Manipulation via Spatiotemporal-Aware View Synthesis
- **标题**：VistaBot: View-Robust Robot Manipulation via Spatiotemporal-Aware View Synthesis
- **简称**：VistaBot
- **领域**：机器人操控 / 跨视角泛化 / 视频生成辅助控制
- **论文**：https://arxiv.org/abs/2604.21914
- **项目**：正文未披露
- **代码**：正文未披露
- **评测**：仿真 benchmark：RLBench 8 个任务，并引入跨视角指标 VGS（View Generalization Score）。真机：Franka FR3 桌面操作。设备配置：Franka FR3 + 平行夹爪；3 路 RealSense D435，相机位姿为 1 路 front 训练视角 + 2 路 side 测试视角；控制频率正文给出约 3 Hz。
- **算力**：正文披露了闭环频率约 3 Hz，但未系统披露训练 GPU 型号、卡数、训练时长、batch size 与显存统计；故记为“正文未披露”。
- **解读**：VistaBot不直接要求多机位标定，而是把测试视角先“翻译”回训练视角，再喂给闭环策略，思路对现有单视角模仿学习/VLA 很友好。亮点是视角鲁棒性明显提升；局限是生成链路带来频率压力与合成伪影风险，复现时应先验证视角还原质量再接控制头。

#### 3) Hi-WM: Human-in-the-World-Model for Scalable Robot Post-Training
- **标题**：Hi-WM: Human-in-the-World-Model for Scalable Robot Post-Training
- **简称**：Hi-WM
- **领域**：机器人后训练 / World Model / 人在环纠错
- **论文**：https://arxiv.org/abs/2604.21741
- **项目**：https://hi-wm.github.io/
- **代码**：正文未披露
- **评测**：仿真 benchmark：正文未采用统一公开仿真 benchmark，而是以 world-model 闭环评估 + 与真实成功率相关性验证为主。真机：3 项双臂操作任务——Fold Towel、Insert L、Route Rope。设备配置：双臂 YAM-Ultra 平台；每臂 6-DoF 动作控制 + gripper；单 top-view 相机；正文未披露 wrist/head 相机。动作执行频率约 15 Hz。
- **算力**：正文详细描述世界模型与 post-training 流程，但未统一披露训练 GPU、卡数、batch、总时长与部署显存；记为“正文未披露”。
- **解读**：Hi-WM把真实机器人上的人在环纠错迁到世界模型里做，允许缓存失败状态、回放分叉并批量采纠错轨迹，工程价值很高。亮点是显著降低硬件占用；局限在于收益强依赖世界模型与真机对齐度，复现时应先验证 rollout 排名与真实成功率相关性，再扩展人类纠错接口。

### B. Agent 工具调用、记忆与工作流系统

#### 1) Tool Attention Is All You Need: Dynamic Tool Gating and Lazy Schema Loading for Eliminating the MCP/Tools Tax in Scalable Agentic Workflows
- **标题**：Tool Attention Is All You Need: Dynamic Tool Gating and Lazy Schema Loading for Eliminating the MCP/Tools Tax in Scalable Agentic Workflows
- **简称**：Tool Attention
- **领域**：Agent / MCP 工具调用 / 上下文工程
- **论文**：https://arxiv.org/abs/2604.21816
- **项目**：正文未单列项目页
- **代码**：https://github.com/asadani/tool-attention
- **评测**：仿真 benchmark：120-tool、6-server 的 MCP/工具调用模拟基准，服务器 token 规模按公开部署审计校准。真机：不适用；设备配置：非机器人论文，无机器人本体/末端/相机位姿。
- **算力**：正文给出核心实现依赖 384 维句向量、FAISS/ChromaDB 检索与两阶段 lazy schema loading；提到嵌入检索延迟可到毫秒级，但未统一披露具体 GPU/CPU 机型、卡数与显存，记为“正文未披露”。
- **解读**：这篇对 MCP/海量 tools 场景很实用：先保留压缩摘要池，再按 query gating 少量加载完整 schema，本质是在工具层做注意力裁剪。亮点是能同时降 token 成本和缩小 tool poisoning 暴露面；局限是效果依赖 summary 质量与阈值校准，复现时要先把工具摘要治理好。

#### 2) StructMem: Structured Memory for Long-Horizon Behavior in LLMs
- **标题**：StructMem: Structured Memory for Long-Horizon Behavior in LLMs
- **简称**：StructMem
- **领域**：Agent Memory / 长时程对话记忆
- **论文**：https://arxiv.org/abs/2604.21748
- **项目**：正文未单列项目页
- **代码**：https://github.com/zjunlp/LightMem
- **评测**：仿真 benchmark：LoCoMo 长期对话记忆基准，并与 RAG、Flat Memory、Graph Memory、多种结构化记忆系统对比。真机：不适用；设备配置：非机器人论文，无机器人本体/末端/相机位姿。
- **算力**：正文披露评测主干使用 gpt-4o-mini 作为问答/评估 backbone，text-embedding-3-small 做 embedding；附录给出检索参数（如检索 60 个 entries 与 5 个 synthesis），但未给 GPU/CPU、显存和总训练时长，记为“正文未披露”。
- **解读**：StructMem把记忆单位从零散事实改为事件级绑定，再做时间锚定与跨事件综合，兼顾多跳推理和维护成本。亮点是比图记忆更省 token/runtime；局限是作者也承认还缺显式冲突更新与遗忘机制，复现时要额外设计 memory update/decay 策略。

#### 3) From Research Question to Scientific Workflow: Leveraging Agentic AI for Science Automation
- **标题**：From Research Question to Scientific Workflow: Leveraging Agentic AI for Science Automation
- **简称**：Scientific Workflow Agent
- **领域**：Agentic AI / 科研自动化 / 工作流编排
- **论文**：https://arxiv.org/abs/2604.21910
- **项目**：正文未单列项目页
- **代码**：https://github.com/hyperflow-wms/1000genome-workflow/tree/main/workflow-composer
- **评测**：仿真 benchmark：1000 Genomes workflow query dataset（150 条自然语言查询，分难度层评测 intent extraction 与 workflow generation）。真机：不适用；设备配置：非机器人论文，无机器人本体/末端/相机位姿。系统部署在 Kubernetes 上完成端到端执行验证。
- **算力**：正文披露端到端执行层面：3 个查询在 Kubernetes 上运行，LLM 额外开销低于 15 秒/query、单 query 成本低于 0.001 美元；Q3 从自然语言到提交完成约 106 秒（11 秒 LLM + 95 秒基础设施）。未披露训练 GPU，因主体并非训练型方法，记为“正文未披露”。
- **解读**：这篇最有价值的是把“LLM 解释意图”和“确定性 DAG 生成”硬分层，并让领域专家用 Skills 文档维护词汇映射与约束，特别适合 OpenClaw/agent skill 体系借鉴。局限是当前验证域较窄，主要在 1000 Genomes；复现时建议先从单域 skill + schema-gated generator 做起。
